## 一、背景与目标

### 1.1 背景

Neko Status 目前已具备完整的设备状态监控、截图上报、服务管理等核心功能。用户群体中存在大量直播场景需求——用户希望通过 Neko Status 直接完成 **SRS 流媒体服务器的推流接入**，无需手动查阅 RTMP 地址、手动配置 OBS，做到"开软件点一下就能播"的极致傻瓜化体验。

### 1.2 目标

| 目标项           | 描述                                                             |
| ---------------- | ---------------------------------------------------------------- |
| **傻瓜化推流**   | 用户无需了解 RTMP 协议，软件自动生成推流配置，一键导入 OBS       |
| **流密钥管理**   | 每台设备拥有唯一 Stream Key，可由服务端统一管理、重置、监控      |
| **SRS 联动**     | 支持用户自托管 SRS 服务器，也支持填写平台提供的 SRS 推流地址     |
| **状态实时反馈** | 推流状态（在线/离线/重试）实时同步到客户端 UI 与服务端后台       |
| **OBS 自动化**   | 通过 OBS WebSocket 协议或 OBS 配置文件导入，减少用户手动操作步骤 |

### 1.3 核心用户故事

```
作为一个用 Neko Status 监控设备的用户，
我希望在软件里直接设置好 SRS 服务器地址，
然后点一个按钮让 OBS 自动配置好推流，
这样我就不用每次手动复制粘贴 RTMP 地址了。
```

## 三、用户交互流程（核心路径）

```
用户首次使用推流功能
        │
        ▼
[设置页 - 直播推流] 填写 SRS 服务器地址 → 测试连通性
        │
        ▼
系统自动生成 Stream Key（或从服务端下发）
        │
        ├── [复制按钮] → 复制 RTMP 推流 URL 到剪贴板 → 手动粘贴到 OBS
        │
        └── [一键配置 OBS 按钮]
                    │
                    ├── 检测 OBS WebSocket 是否可连接
                    │       ├─ 可连接 → 自动写入 OBS 推流配置 → 提示"OBS 已就绪"
                    │       └─ 不可连接 → 生成 obs-service.json 文件 → 指引用户手动导入
                    │
                    ▼
            [推流控制页] 显示推流状态（RTMP 心跳检测 / SRS on_publish 回调）
```

## 四、技术选型决策

### 4.1 OBS 自动化方案对比

| 方案                             | 优点                     | 缺点                              | 决策         |
| -------------------------------- | ------------------------ | --------------------------------- | ------------ |
| OBS WebSocket（`obs-websocket`） | 全自动，无需用户手动操作 | 用户需提前安装 OBS WebSocket 插件 | **主方案**   |
| OBS 服务配置文件导入（JSON）     | 无依赖，兼容性强         | 用户需手动在 OBS 中导入一次       | **兜底方案** |
| 纯手动复制粘贴                   | 零依赖                   | 体验差，存在失误风险              | 仅显示辅助   |

> **结论**：优先尝试 OBS WebSocket 自动化，失败时降级为 JSON 文件导出引导，始终提供手动复制兜底。

### 4.2 推流状态检测方案

| 方案                | 说明                                          | 决策            |
| ------------------- | --------------------------------------------- | --------------- |
| SRS HTTP API 轮询   | 定期调用 `GET /api/v1/streams` 检查流是否在线 | **主方案**      |
| SRS on_publish 回调 | SRS 推流开始时主动 POST 到 Neko 后端          | **v1.2.0 实现** |
| RTMP 握手主动探测   | 主进程尝试 TCP 连接 RTMP 端口检测服务可用性   | 连通性测试用    |

### 4.3 Stream Key 管理策略

- **格式**：`nk_{device_id}_{8位随机hex}` ，例如 `nk_dev_abc123_f3e2d1c0`
- **存储**：服务端 DB 持久化，客户端 `ConfigStore` 缓存
- **重置**：用户主动触发，服务端重新生成，旧 Key 立即失效

## 六、非功能性要求

| 分类         | 要求                                                                    |
| ------------ | ----------------------------------------------------------------------- |
| **安全**     | Stream Key 不得明文出现在日志、console 输出中；传输层必须支持 RTMPS/TLS |
| **可靠性**   | OBS WebSocket 连接失败不得 crash 应用；用优雅降级提示代替报错弹窗       |
| **性能**     | 推流状态轮询间隔不少于 10s，避免对 SRS 产生不必要的查询压力             |
| **离线兼容** | 用户未配置 SRS 服务器时，推流页面应显示引导步骤而非空白/报错            |
