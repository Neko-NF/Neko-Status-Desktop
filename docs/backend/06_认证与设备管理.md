# 06 — 认证与设备管理

---

## 一、认证模型概述

Neko Status 采用 **API Key 认证**方案：

- 每台设备首次运行时自动向服务端注册，获取唯一 API Key
- API Key 格式：`nk_stat_` + 24 位随机字符，共 32 位
- 所有受保护接口在请求头携带 `X-API-Key`
- 服务端**不存明文**，只存 `SHA256(api_key)` 的哈希值

---

## 二、API Key 生成规范

### 格式

```
nk_stat_[24位随机字符（大小写字母+数字）]
```

示例：`nk_stat_A3xK9mPqR7vB2nLdT5wE6cFj`

### 生成代码（Node.js）

```javascript
const crypto = require("crypto");

function generateApiKey() {
  // 生成 18 字节随机数，Base62 编码后约 24 位
  const random = crypto.randomBytes(18).toString("base64url").slice(0, 24);
  return `nk_stat_${random}`;
}

function hashApiKey(apiKey) {
  return crypto.createHash("sha256").update(apiKey).digest("hex");
}
```

### 验证流程

```javascript
async function verifyApiKey(providedKey) {
  if (!providedKey || !providedKey.startsWith("nk_stat_")) {
    return null; // 格式错误
  }

  const keyHash = hashApiKey(providedKey);

  const apiKey = await db.query(
    `SELECT ak.*, d.id AS device_id, d.is_active
     FROM api_keys ak
     JOIN devices d ON ak.device_id = d.id
     WHERE ak.key_hash = $1
       AND ak.revoked_at IS NULL
       AND (ak.expires_at IS NULL OR ak.expires_at > NOW())
       AND d.is_active = TRUE`,
    [keyHash],
  );

  if (!apiKey) return null;

  // 更新最近使用时间（异步，不阻塞请求）
  db.query("UPDATE api_keys SET last_used_at = NOW() WHERE key_hash = $1", [
    keyHash,
  ]);

  return apiKey;
}
```

## 三、认证中间件

```javascript
// middleware/auth.js

async function authMiddleware(req, res, next) {
  const apiKey = req.headers["x-api-key"];

  if (!apiKey) {
    return res.status(401).json({
      ok: false,
      error: { code: "UNAUTHORIZED", message: "缺少 X-API-Key 请求头" },
    });
  }

  const keyRecord = await verifyApiKey(apiKey);

  if (!keyRecord) {
    return res.status(401).json({
      ok: false,
      error: {
        code: "INVALID_API_KEY",
        message: "API Key 无效、已过期或已被吊销",
      },
    });
  }

  // 将设备信息挂到 req 对象
  req.deviceId = keyRecord.device_id;
  req.apiKeyId = keyRecord.id;

  next();
}

// 本地测试模式：跳过认证（仅当服务器运行在本地且配置了 ALLOW_NO_AUTH=true）
async function optionalAuthMiddleware(req, res, next) {
  if (
    process.env.ALLOW_NO_AUTH === "true" &&
    process.env.NODE_ENV !== "production"
  ) {
    req.deviceId = "local-test-device";
    return next();
  }
  return authMiddleware(req, res, next);
}

module.exports = { authMiddleware, optionalAuthMiddleware };
```

> **安全说明**：`ALLOW_NO_AUTH` 仅允许在本地开发环境中使用，**绝对不允许在生产环境开启**。

## 四、设备注册流程

### 流程图

```
客户端首次启动
      │
      ▼
本地是否已有 API Key？
  │          │
 是          否
  │          │
  ▼          ▼
直接使用   POST /api/v1/auth/register
              │
         设备 MAC 哈希已注册过？
              │
        ┌─────┴─────┐
      是（重复）   否（新设备）
        │              │
    返回原有          创建 devices 记录
    设备 API Key      生成新 API Key
    （可选策略）      写入 api_keys 表
                       │
                   返回 api_key（仅此一次）
                       │
                  客户端持久化到本地
                  （Electron safeStorage）
```

### 防重复注册策略

同一台机器（相同 `mac_hash`）多次注册时，服务端策略：

| 方案                   | 说明                                 | 推荐                 |
| ---------------------- | ------------------------------------ | -------------------- |
| 返回已有 Key           | 幂等注册，客户端重装后仍能使用原 Key | ✅ Beta 阶段推荐     |
| 创建新 Key，吊销旧 Key | 每次重装获得新 Key                   | 可选                 |
| 拒绝重复注册           | 需人工解绑                           | 不推荐（用户体验差） |

## 五、API Key 吊销

### 主动吊销接口

```
DELETE /api/v1/auth/api-key
X-API-Key: nk_stat_xxx（当前 key）
```

**Response 200**

```json
{ "ok": true, "message": "API Key 已吊销，设备将需要重新注册" }
```

**后端逻辑：**

```sql
UPDATE api_keys
SET revoked_at = NOW()
WHERE key_hash = $1;
```

### 管理员批量操作（可选，P2）

```sql
-- 吊销某设备的所有 Key
UPDATE api_keys SET revoked_at = NOW()
WHERE device_id = 'dev_abc123' AND revoked_at IS NULL;

-- 禁用设备
UPDATE devices SET is_active = FALSE WHERE id = 'dev_abc123';
```

## 六、本地测试模式

客户端支持"本地测试"模式（在设置的配置弹窗中切换），此模式下：

- 服务器地址：`http://localhost:8080`
- **无需 API Key**（服务端开启 `ALLOW_NO_AUTH=true`）
- 主要用于开发调试，不会向远程服务发送真实数据

**客户端判断逻辑：**

```javascript
const isLocalTest =
  serverUrl.startsWith("http://localhost") ||
  serverUrl.startsWith("http://127.0.0.1");

const headers = {
  "Content-Type": "application/json",
  ...(isLocalTest ? {} : { "X-API-Key": settings.get("apiKey") }),
};
```

## 七、安全注意事项

| 威胁                   | 防护措施                                                             |
| ---------------------- | -------------------------------------------------------------------- |
| API Key 泄露           | 不存明文；客户端用 `electron.safeStorage` 加密存储                   |
| Stream Key 泄露        | 日志中掩码处理（`nk_dev_****`）；使用 `crypto.randomBytes()` 生成    |
| OBS WebSocket 密码泄露 | 使用 Electron `safeStorage` 加密存储，不得以明文写入 `config.json`   |
| 暴力破解               | 登录失败限速（每 IP 每小时 10 次），Key 格式检查（`nk_stat_` 前缀）  |
| Stream Key 枚举        | Key 重置接口频率限制（每设备每分钟最多 3 次）                        |
| 中间人攻击             | 生产环境强制 HTTPS/WSS；推流建议使用 RTMPS（端口 443）               |
| SSRF（自定义更新源）   | 见 05 文档，严格校验 GitHub repo 格式                                |
| SSRF（SRS 代理接口）   | `test-srs` / `status` 接口需校验目标地址为合法 IP/域名，禁止回环地址 |
| 日志泄露               | 日志中只记录 `key_prefix`（前8位），不记录完整 Key                   |
| SQL 注入               | 所有数据库操作使用参数化查询，禁止字符串拼接 SQL                     |

## 八、速率限制实现（参考）

```javascript
// 使用 express-rate-limit
const rateLimit = require("express-rate-limit");

const apiLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 分钟窗口
  max: 120, // 一般接口：120次/分钟
  keyGenerator: (req) => req.deviceId || req.ip, // 按设备ID限流
  message: {
    ok: false,
    error: { code: "RATE_LIMITED", message: "请求过于频繁，请稍后重试" },
  },
});

const statusLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 60, // 状态上报：60次/分钟
  keyGenerator: (req) => req.deviceId,
});

const updateLimiter = rateLimit({
  windowMs: 60 * 60 * 1000, // 1小时窗口
  max: 10, // 更新检查：10次/小时
  keyGenerator: (req) => req.deviceId,
});

const streamKeyResetLimiter = rateLimit({
  windowMs: 60 * 1000, // 1分钟窗口
  max: 3, // Stream Key 重置：3次/分钟
  keyGenerator: (req) => req.deviceId,
  message: {
    ok: false,
    error: {
      code: "RATE_LIMITED",
      message: "Stream Key 重置过于频繁，请稍后重试",
    },
  },
});

app.use("/api/", apiLimiter);
app.use("/api/v1/status", statusLimiter);
app.use("/api/v1/update", updateLimiter);
app.use("/api/v1/stream/key/reset", streamKeyResetLimiter);
```

## 九、Stream Key 管理（v1.1.0 新增）

> 关联文档：《SRS直播推流-后端实现方案.md》、《02_REST_API接口文档.md》第六节

### Stream Key 与 API Key 的关系

| 维度     | API Key                          | Stream Key                      |
| -------- | -------------------------------- | ------------------------------- |
| 用途     | REST API / WebSocket 身份认证    | SRS RTMP 推流鉴权               |
| 格式     | `nk_stat_` + 24位随机字符        | `nk_{deviceId前8位}_{8位hex}`   |
| 存储     | 服务端仅存 SHA256 哈希           | 服务端明文存储（需与 SRS 匹配） |
| 数量     | 一设备可有多个（吊销后重新注册） | 一设备一个（重置替换）          |
| 生命周期 | 注册时生成，可吊销               | 首次调用时生成，可重置          |

### 生成规范

```javascript
const crypto = require("crypto");

function generateStreamKey(deviceId) {
  const prefix = deviceId.replace(/[^a-zA-Z0-9]/g, "").substring(0, 8);
  const rand = crypto.randomBytes(4).toString("hex");
  return `nk_${prefix}_${rand}`;
}
```

### 重置策略

- 用户主动触发 `POST /api/v1/stream/key/reset`
- 旧 Key 立即失效（数据库覆盖写入）
- 频率限制：每设备每分钟最多 3 次
- 重置后客户端需重新配置 OBS 推流地址

### 日志脱敏

```javascript
// 所有日志输出必须使用掩码函数
function maskStreamKey(key) {
  if (!key || key.length < 10) return "***";
  return key.substring(0, 6) + "****";
}
// 输出示例：nk_dev_****
```
